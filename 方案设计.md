# 基于 RK3588 的 AI 视觉+传感融合边缘智能网关 — 技术方案

## 项目定位

面向工业现场两大痛点——"老旧设备协议碎片化"与"视频数据价值密度低"——基于 RK3588 国产 AIoT 旗舰芯片构建的边缘智能融合网关。核心价值：**在边缘侧完成 AI 推理，实现视频与传感器数据的事件级融合，让传统哑设备获得智能化改造能力。**

区别于传统网关"只做协议转换"和 AI 推理"纯云端处理"，本项目在单一硬件平台上完成 **Modbus 传感器采集 → RTSP 视频流硬解码 → NPU 端侧推理 → 多源事件融合 → 本地异步存储 → MQTT 标准化上云** 的完整闭环。

## 系统架构全景

### 物理架构：五进程 + 四层逻辑

```
┌──────────────────────────────────────────────────────────────────┐
│                      云端 MQTT Broker (SparkplugB)                 │
│                     QoS1 + 遗嘱 + 配置下发 + OTA                   │

│  │ └──────────┘ └───────────┘ └────────────────────┘ │         │
│  │                                                    │         │
│  └────────────────────────────────────────────────────┘         │
└──────────────────────────────────────────────────────────────────┘
```

**核心架构原则**：

1. **原始视频帧不跨进程**：视频解码、预处理、NPU 推理全在进程 E 内部完成，跨进程只传小体量结构化检测结果（框坐标+类别+置信度+时间戳），解决多进程架构 IPC 大块数据开销问题。
2. **不稳定模块物理隔离**：GStreamer 管线、RKNN 驱动、Modbus 串口阻塞——任一崩溃由 Watchdog 独立拉起，不影响其他进程。
3. **业务逻辑收敛在进程 B**：所有规则判断、事件融合、存储决策、北向发送集中在一处，保证数据一致性和事务性。

### 逻辑分层（四层模型映射到进程）

| 逻辑层 | 物理进程 | 职责 |
|--------|---------|------|
| **接入层** (South Driver) | 进程 A | 屏蔽硬件差异，所有南向数据统一为 InternalMessage |
| **核心层-AI** (Edge AI) | 进程 E 🆕 | 视频解码 + NPU 推理，输出结构化检测结果 |
| **核心层-业务** (Edge Fusion) | 进程 B | 规则引擎、事件融合、存储编排、北向通信 |
| **展示层** (Monitor) | 进程 C | Qt 仪表盘可视化，不参与任何 IO |

> 数据层（SQLite）和北向层（MQTT/OTA）在逻辑上是独立层，但在物理上收敛在进程 B 内部——因为这两层与规则引擎紧密耦合，拆分反而增加 IPC 复杂度和事务一致性风险。

## 核心模块设计

### 进程 A：南向接入网关 (gateway_access)

**职责**：所有南向数据的统一入口，屏蔽 Modbus RTU、Modbus TCP、GPIO、I2C 等硬件接口差异，归一化为 InternalMessage 经 UDS 发给进程 B。

#### 驱动架构：可插拔适配器

```cpp
// 每个南向驱动实现此接口
class ISensorDriver {
public:
    virtual ~ISensorDriver() = default;
    virtual void start() = 0;
    virtual void stop() = 0;
    virtual std::string_view name() const = 0;

    // 回调：驱动采集到数据后调用，传入归一化后的 InternalMessage
    std::function<void(const InternalMessage&)> on_data;
};
```

**MVP 阶段**：编译期多态（`vector<unique_ptr<ISensorDriver>>` 注册），不做动态 .so 加载。

**M2 扩展**：通过配置文件指定 .so 路径 + `dlopen/dlsym` 工厂函数，新驱动无需重编译主程序。

#### Modbus RTU 驱动（复用旧方案核心逻辑）

保留旧方案的手写 Modbus RTU 协议栈（~300 行），核心设计不变：

- **轮询调度器**：半双工 RS485 总线时间片调度，`for each device: 发请求 → 等响应(500ms超时) → CRC校验 → 解析`
- **超时剔除与复活**：连续 3 次超时挂起设备、30s 尝试复活一次
- **eventfd 异步桥接**：Modbus 线程阻塞读串口 → 封装 InternalMessage → write(eventfd) 唤醒主 epoll 线程 → UDS 转发
- **真机验证**：RK3588 两路 UART 物理交叉回环（替代原 i.MX6ULL），PC 端 socat 降级

#### Modbus TCP 驱动 🆕

以太网上的 Modbus 变体（端口 502），为以太网 PLC 和智能电表预留。与 RTU 驱动的区别：传输层是 TCP 非串口（无半双工限制，可并发多连接），帧格式去掉 CRC（TCP 已保证传输完整性）但保留 MBAP 头。MVP 阶段不实现，M2 通过 `ModbusTcpDriver` 加入。

#### GPIO / I2C 驱动 🆕

- **GPIO**：sysfs 接口操作板载 LED/继电器，规则引擎触发报警时驱动硬件动作
- **I2C**：读取板载传感器（需确认 RK3588 开发板原理图），数据封装为 InternalMessage

#### 关键设计决策

| 决策 | 选型 | 原因 |
|------|------|------|
| Modbus 协议栈 | **手写** (~300行) | 帧结构简单，面试可讲清每个字节；交叉编译少一个依赖 |
| 动态插件 | M1 不做，M2 加入 | M1 跑通主链路，M2 用 dlopen 展示工程扩展性 |
| RS485 半双工 | 独立线程 + eventfd | 串口阻塞不能放 epoll 主循环，老方案已验证 |

### 进程 E：AI 推理引擎 (gateway_engine) 🆕

**职责**：从 RTSP 摄像头拉流 → GStreamer VPU 硬解码 → 预处理 → RKNN NPU 推理 → 后处理 → 结构化检测结果经 UDS 发给进程 B。这是新方案与旧方案的**本质差异**。

#### 为什么独立进程

GStreamer 管线、RKNN 驱动均为第三方 C 库，稳定性不可控。一旦 segfault，不能拖死 Modbus 采集和 MQTT 上云。进程隔离是最低成本的保障手段。

#### 异步流水线设计

采用 **Pipeline + Stage 抽象模式**，四级流水线各封装为独立 Stage 类，由 Pipeline 编排器统一管理生命周期、SPSC 队列和帧内存池。

```
┌──────────────────────────────────────────────────────┐
│                  Pipeline (编排类)                    │
│  - 创建 3 条 SPSC 队列 + FramePool 帧内存池           │
│  - 管理 4 个 Stage 的启停顺序（倒序启动，正序停止）      │
│  - 模型热切换：销毁旧 InferenceStage → 重建新           │
├──────────────────────────────────────────────────────┤
│                                                      │
│  ┌─────────────┐   SPSC队列   ┌──────────────┐       │
│  │ GstCaptureStage│ ──────────→ │PreprocessStage│      │
│  │ GStreamer    │             │ resize       │       │
│  │ RTSP 硬解    │             │ letterbox    │       │
│  │ NV12 输出    │             │ (NV12原样透传) │       │
│  └─────────────┘             └──────┬───────┘       │
│                                     │ SPSC队列      │
│                          ┌──────────┴──────────┐    │
│                          │ RknnInferenceStage   │    │
│                          │ rknn_run (同步调用)   │    │
│                          └──────────┬──────────┘    │
│                                     │ SPSC队列      │
│                          ┌──────────┴──────────┐    │
│                          │ PostprocessStage     │    │
│                          │ NMS/解码/JPEG截图    │    │
│                          │ → DetectionResult    │    │
│                          │ → UDS → 进程B        │    │
│                          └─────────────────────┘    │
│                                                      │
├──────────────────────────────────────────────────────┤
│  网关特有层: 按需拉流调度 / 业务心跳 / cgroups 隔离     │
└──────────────────────────────────────────────────────┘
```

**Stage 启停顺序（关键设计）**：

| 操作 | 顺序 | 原因 |
|------|------|------|
| **启动** | Postprocess → Inference → Preprocess → Capture（倒序） | 消费者先就位，确保生产者入队时下游已准备好消费 |
| **停止** | Capture → Preprocess → Inference → Postprocess（正序） | 源头先停，下游消费完队列中剩余数据后自然退出 |

**FramePool 帧内存池**：
- Pipeline 初始化时预分配 6 个帧缓冲区（`config.frame_pool_size = 6`）
- Stage 间传递 `shared_ptr<Frame>`，用完自动归还
- 避免每帧 malloc/free，零动态内存分配

**SPSC 队列容量**：
- 每级 SPSC 队列容量 `config.queue_capacity = 4`（FramePool 共 6 帧，4 帧在队列中可保证流水线不阻塞）
- 溢出策略：Capture→Preprocess 队列满时**丢弃最旧帧**（保证推理最新画面）；下游队列满时**生产者阻塞等待**（推理结果不可丢弃）

#### 按需拉流机制 🆕

不是 7×24 持续拉流。规则引擎默认不触发分析任务时，进程 E 处于待机状态（不建立 RTSP 连接）。触发条件支持两种模式：

- **事件驱动模式**（M3 默认）：传感器检测到异常（如区域入侵事件）→ 规则引擎判定需要视觉确认 → 通过 UDS 下发 START_ANALYSIS 指令
- **时间表模式**（M4 扩展）：配置文件定义定时巡检计划（如每小时巡检 5 分钟）

当进程 B 下发的"分析任务"到达时，进程 E 才建立 RTSP 连接开始拉流。任务完成后断开连接，节省网络带宽和 NPU 资源。

```
进程 B → UDS → 进程 E: "START_ANALYSIS camera=zone_A model=yolov8n"
进程 E: GStreamer 建立 RTSP → 推理 → 发送 DetectionResult → ...
进程 B → UDS → 进程 E: "STOP_ANALYSIS camera=zone_A"
进程 E: 断开 RTSP，进入待机
```

#### 模型热切换

- 首次启动加载默认模型 `yolov8n.rknn`
- 收到进程 B 的 `SWITCH_MODEL` 指令 → 校验新模型文件 MD5 → `rknn_destroy` 释放旧 context → `rknn_init` 加载新模型 → 回复进程 B ACK
- 切换期间队列中的帧直接丢弃（不做推理），切换完成后恢复正常
- **失败回滚**：`rknn_init` 失败 → 重新 `rknn_init` 旧模型 → 回复进程 B NACK + 错误原因 → 进程 B 发布 MQTT 告警。若旧模型也加载失败（模型文件损坏等极端情况）→ 进程 E 上报 FATAL → Watchdog kill -9 后拉起恢复默认模型

#### 稳定性设计

| 机制 | 实现 |
|------|------|
| **业务心跳** | 每 5s 通过 UDS 向进程 B 发送心跳（含当前 FPS、NPU 温度） |
| **自动重启** | 进程 B 30s 未收到心跳 → 判定 E 异常 → Watchdog kill -9 + 重启 |
| **RTSP 重连** | 断连后指数退避重试（1s→2s→4s→...→最大 30s） |
| **NPU 过热保护** | 读取 NPU 温度（路径以 RK3588 实际 sysfs 为准，通常为 `/sys/class/npu/npu_temp` 或 thermal_zone），>85°C 时自动降帧率 |
| **资源隔离** | cgroups v1（RK3588 默认内核支持）限制 CPU ≤ 60%、内存 ≤ 2GB，确保 Modbus 和 MQTT 永远有资源 |

#### 关键设计决策

| 决策 | 选型 | 原因 |
|------|------|------|
| 推理框架 | RKNN（瑞芯微官方） | RK3588 NPU 唯一官方推理 API |
| 视频解码 | GStreamer + VPU 硬解 | 硬件解码 NV12，零 CPU 开销 |
| 预处理 | RKNN 内置 `rknn_input_set` | 支持直接传 NV12，省去 CPU 端色彩空间转换（YUV→RGB）；letterbox/resize 等几何变换仍在 PreprocessStage 完成 |
| 目标检测模型 | YOLOv8n（后续扩展 YOLOv11） | 6TOPS NPU 可实时推理 |
| 按需拉流 | B 通过 UDS 反向指令触发 | 节省带宽 + NPU 资源 |

### 进程 B：业务中枢 (gateway_core)

**职责**：事件融合 + 规则引擎 + SQLite 异步存储 + MQTT 上云 + 断网续传 + OTA 管理 + 配置热加载 + 共享内存发布。这是整个网关的"大脑"。

**自身 epoll 循环**，监听六个 fd：
- UDS server fd（进程 A 报文流入）
- UDS server fd（进程 E 检测结果流入）🆕
- MQTT 内部 socket fd
- eventfd（MQTT 回调线程唤醒）
- eventfd（A 和 E 共用数据通知，由 UDS 接收线程批量写入后触发）🆕
- 信号 fd（SIGINT/SIGTERM 优雅退出）

#### 事件融合器 🆕（核心差异化模块）

旧方案的规则引擎只做阈值判断（温度>80→报警）。新方案的核心升级是**多源事件融合**——将 AI 检测结果和传感器读数在时间+空间维度上关联：

```
AI 检测: 安全帽缺失 @ 区域A, 置信度 0.92
    ↓ 立即查询
传感器: ZONE_A_TEMP = 42.5°C, MOTOR_A_STATUS = "Running"
    ↓ 融合组装
告警事件: {
    "type": "SAFETY_VIOLATION",
    "subtype": "NO_HELMET",
    "location": "ZONE_A",
    "ai": {"class": "no_helmet", "conf": 0.92, "snapshot_jpeg": 200KB},
    "context": {"temperature": 42.5, "motor_running": true},
    "severity": "HIGH",  // 高温+安全违规 → 升级严重度
    "ts": 1718123456
}
```

融合规则表：

| AI 事件 | 传感器条件 | 融合告警等级 | 动作 |
|---------|-----------|------------|------|
| 安全帽缺失 | 任意 | HIGH | SQLite + MQTT + JPEG 快照 + GPIO 声光报警 |
| 区域入侵 | 时间段=非工作时间 | CRITICAL | SQLite + MQTT QoS1 + 快照 + 继电器触发 |
| 区域入侵 | motor_running=true | MEDIUM | SQLite + MQTT（不触发外设） |
| 烟火检测 | 任意 | CRITICAL | SQLite + MQTT + fsync 强制刷盘 + GPIO 消防联动 |
| 设备异常高温 | AI 检测到设备区域有人员 | HIGH | SQLite + MQTT |
| 设备异常高温 | AI 检测到设备区域无人员 | MEDIUM | SQLite + MQTT |

#### 规则引擎（从旧方案复用并增强）

保留旧方案的四条基础规则（高温报警、视觉入侵、传感器离线、周期存储），新增：

| 规则 | 条件 | 动作 |
|------|------|------|
| 多源复合告警 | 事件融合器输出 | SQLite + MQTT + 共享内存 |
| AI 引擎心跳超时 | 30s 无心跳 | 触发 Watchdog 重启进程 E |
| NPU 过热降频 | 温度 > 85°C | MQTT 上报 + 降低分析任务频率 |
| 按需拉流触发 | 时间表/事件驱动 | 通过 UDS 反向指令启停进程 E |

#### SQLite 异步批量落盘（直接复用旧方案）

独立 DB 线程 + 无锁 SPSC 队列 + 事务批量提交（每 100ms 或 攒够 50 条）。业务线程零阻塞。详细设计见旧方案，核心要点不变：
- `PRAGMA journal_mode=WAL; synchronous=NORMAL;`
- 关键报警事务后手动 fsync() 强制刷盘
- sensor_data 表 + alarm_log 表 + offline_cache 表

#### MQTT 北向通信

从旧方案的自定义 Topic 升级为 **部分 SparkplugB 兼容**：

| 旧方案 Topic | 新方案 Topic | 变化 |
|-------------|-------------|------|
| `gateway/gw001/sensor` | `spBv1.0/edge_gateway/DDATA/gw001/sensor` | 采用 SparkplugB 命名规范 |
| `gateway/gw001/alarm` | `spBv1.0/edge_gateway/DALARM/gw001` | 告警独立主题 |
| `gateway/gw001/status` (遗嘱) | `spBv1.0/edge_gateway/DBIRTH/gw001` + Will | BIRTH/DEATH 语义 |
| `gateway/gw001/rpc` | `spBv1.0/edge_gateway/DCMD/gw001` | RPC 指令 |

> 不完全遵循 SparkplugB 全部规范（Payload 不强制 Protobuf 编码，MVP 阶段保留 JSON 便于调试；M4 可选升级 Protobuf 实现完全兼容，利用二进制编码的高效性和类型安全），但在 Topic 命名上对齐行业标准。

**保留旧方案的全部 MQTT 特性**：QoS1、Last Will 遗嘱、MQTT 回调线程隔离（禁止回调中写 SQLite/做 UDS 转发）。

#### 断网续传（直接复用）

MQTT 断连 → 写入 `offline_cache` 表 → 恢复后按优先级（报警>周期数据）异步补传 QoS1 → 成功后 DELETE。直接复用旧方案设计。

#### 远程 RPC + 配置热加载 🆕（旧方案 RPC 的升级版）

旧方案 RPC 只能改个别参数且需 A/B 进程间转发。新方案升级为完整的配置热加载——整个 JSON/YAML 配置文件通过 MQTT 下发、校验、原子替换、通知各模块重载，**全程不重启进程**：

```
MQTT DCMD 收到 config_update:
  1. 校验 JSON 格式 → 失败则回复 NACK + 错误原因
  2. 备份旧配置: cp system.json system.json.bak
  3. 写入新配置: 原子 write(tmp) + rename
  4. 逐模块通知重载:
     - Modbus 轮询间隔/地址表 → UDS 通知进程 A
     - AI 分析规则/阈值 → UDS 通知进程 E
     - 内部参数（报警阈值等）→ 进程 B 自身直接生效
  5. 回复 ACK + 新配置 MD5
```

支持的 RPC 指令扩展：

| method | 目标进程 | 说明 |
|--------|---------|------|
| `set_modbus_interval` | A | 动态修改 Modbus 轮询间隔 |
| `set_alarm_threshold` | B | 动态修改报警阈值 |
| `add_modbus_device` | A | 热添加 Modbus 从站 |
| `start_analysis` | E 🆕 | 触发按需拉流 |
| `stop_analysis` | E 🆕 | 停止拉流 |
| `switch_model` | E 🆕 | 切换 AI 模型 |
| `trigger_offline_sync` | B | 手动触发断网缓存补传 |
| `get_device_health` | B | 获取设备体检报告 |
| `ota_update_model` | B→E | 模型 OTA 升级 |

安全措施（与旧方案一致）：method 白名单 + id 防重放 + 参数范围校验。商业化落地时加入 HMAC-SHA256 签名 + TLS。

#### 模型 OTA 🆕（全新模块）

进程 B 负责 OTA 编排，进程 E 负责模型加载：

```
MQTT DCMD 收到 ota_update_model:
  1. 进程 B 从指定 URL 分块下载 .rknn 模型文件（断点续传）
  2. 下载完成后校验 SHA256
  3. 通知进程 E：准备切换模型
  4. 进程 E：rknn_destroy → rknn_init(新模型)
  5. 成功 → 进程 E 回 ACK → 进程 B 记录新版本号到 SQLite
  6. 失败 → 进程 E 自动回滚旧模型 → 进程 B 收到 NACK → 发布告警
```

### 进程 B 内部通信：UDS 协议

进程 A → B 和进程 E → B 均使用 Unix Domain Socket（`SOCK_STREAM`），统一 InternalMessage 格式（复用旧方案 13 字节定长头 + 变长体）：

| 字段 | 说明 |
|------|------|
| `source_type` (4B) | 0=Modbus, 1=GPIO, 2=I2C, 3=AI检测 🆕 |
| `node_id` (4B) | 来源节点/设备 ID |
| `tlv_type` (1B) | 0x01=温湿度, 0x04=AI视觉告警, 0x05=AI心跳 🆕, 0xFF=设备心跳 |
| `payload_len` (4B) | payload 字节数 |
| `payload` (变长) | 具体数据 |

## 共享内存：Ping-Pong 双缓冲（直接复用旧方案）

完整复用旧方案的 Ping-Pong 双缓冲设计，核心不变：

- **ShmRegion**: 双缓冲数组 `buffer[0]` / `buffer[1]` + `std::atomic<uint32_t> read_index`，写端写 `buffer[1 - read_index]`（备用区），完成后原子切换 `read_index`，读端只读 `buffer[read_index]`。
- **读写物理隔离**：从根本上消除数据撕裂

**ShmBlock 字段更新（新方案新增 AI 相关字段）**：

```cpp
struct ShmBlock {
    // ── 网关基础指标 (复用) ──
    int64_t  uptime_sec;
    int32_t  total_packets;
    int32_t  total_alarms;
    float    cpu_usage;
    float    mem_usage;
    int32_t  online_nodes;
    int32_t  mqtt_connected;
    int32_t  alarm_active;
    char     last_alarm[128];

    // ── AI 引擎指标 🆕 ──
    float    npu_temp_c;           // NPU 温度
    float    inference_fps;         // 推理帧率
    int32_t  ai_engine_online;     // 进程 E 是否在线 (0/1)
    int32_t  model_version;        // 当前加载的模型版本号
    uint32_t last_detection_ts;    // 最后一次检测的时间戳
    char     last_model_name[64];  // 当前模型名称

    // ── AI 检测快照图片数据 🆕 ──
    int32_t  snapshot_jpeg_len;    // JPEG 快照字节数，0=无
    // ⚠️ JPEG 体积 (100-300KB) 不入 ShmBlock。进程 B 通过 UDS
    // 直接向进程 C 推送完整 JPEG（UDS send buffer ≥ 512KB）。
    // snapshot_jpeg_len >0 仅做触发信号，C 收到后通过 UDS 接收。

    uint8_t  _reserved[128];
};
static_assert(sizeof(ShmBlock) <= 4096, "ShmBlock 超出页大小");
```

> ⚠️ **JPEG 快照传递方案**：JPEG 体积 (100-300KB) 不适合放共享内存。进程 B 通过 UDS 直接向进程 C 推送完整 JPEG 数据（需确保 UDS send buffer ≥ 512KB）。`snapshot_jpeg_len > 0` 仅做触发信号，C 检测后通过 UDS 接收 JPEG。此方案避免磁盘 IO，不消耗 eMMC 写寿命。

## 进程看门狗 (gateway_watchdog)

**职责**：直接复用旧方案设计——fork + waitpid 循环守护 A/E/B/C 四个子进程。核心策略不变：

| 策略 | 说明 |
|------|------|
| 崩溃重拉 | 子进程异常退出后等待 1s 再拉起 |
| 重启退避 | 30s 内重启超 5 次 → 暂停重启 + FATAL 日志 |
| 依赖顺序 | 先启动 B → 等待 B 就绪信号 → 启动 E + A → 等待就绪 → 最后启动 C |
| 日志隔离 | 每个子进程独立日志文件 |
| systemd 兜底 | Watchdog 自身 `Restart=always` |

🆕 **进程就绪信号机制**：所有子进程启动后通过 UDS 向进程 B 发送 READY 信号。Watchdog 等待就绪信号后再进入正常监控周期。进程 E 若 10s 内未收到 READY → 判定启动失败 → kill -9 → 重试（计入重启退避计数）。进程 A 同理。

**进程 B 重启后的状态恢复**（复用旧方案）：扫描 SQLite 中重启期间的记录 → 回填共享内存 last_alarm → 重建 MQTT 连接 → 补发 BIRTH 消息 → 通知进程 E 恢复分析任务。

**技术栈**：`C++17` `RK3588` `RKNN` `GStreamer` `Modbus RTU/TCP` `MQTT SparkplugB` `SQLite3` `Qt5/QPainter` `多进程IPC` `epoll` `共享内存` `cgroups`

**项目描述**：
> 面向工业现场设备协议碎片化与视频数据利用率低两大痛点，基于国产瑞芯微 RK3588 旗舰芯片（6TOPS NPU）构建边缘智能融合网关。南向支持 Modbus RTU/TCP 多协议传感器接入与 RTSP 摄像头 VPU 硬解码；核心层实现 YOLOv8 端侧 NPU 推理与传感器-AI 多源事件融合；北向 MQTT SparkplugB 标准化上云，含断网续传、远程 RPC、配置热加载与模型 OTA。五进程架构配合 Watchdog 守护 + cgroups 资源隔离，满足工业现场 7×24 无人值守高可用需求。

**技术亮点**：

- **端侧 AI 推理异步流水线**：GStreamer VPU 硬解码 RTSP 流（NV12 DMA-BUF 直传 NPU，免 CPU 色彩空间转换）→ RKNN NPU 推理 → 后处理，解码/预处理/推理/后处理四级流水 + SPSC 无锁队列线程模型，6TOPS NPU 上 YOLOv8n 实时推理 >25fps

- **多源事件融合引擎**：设计 AI 检测结果与传感器读数的时空关联规则表（6 类复合告警逻辑），安全违规+高温→自动升级严重度并联动 GPIO 声光报警，实现从单源阈值告警到多源决策级融合的升级

- **模型热切换与 OTA**：支持 MQTT 远程下发模型文件（分块下载 + SHA256 校验）→ `rknn_destroy/init` 热切换 NPU 模型→失败自动回滚，全程推理服务不中断

- **南向多协议归一化**：基于 ISensorDriver 适配器接口 + dlopen 动态插件机制，手写 Modbus RTU/TCP 协议栈（CRC16/MBAP/超时剔除与复活），Modbus 与 GPIO/I2C 数据内部归一化为统一 InternalMessage 格式

- **配置全热加载**：MQTT 下发 JSON 配置文件 → MD5 校验 → 原子写入 + rename → 逐模块通知重载（Modbus 轮询间隔、AI 分析阈值、报警规则），全流程不重启进程

- **进程看门狗 + cgroups 隔离**：五进程 fork/waitpid 守护 + 指数退避重启 + systemd 双重兜底；AI 推理进程 cgroups 限制 CPU ≤ 60% 内存 ≤ 2GB，确保 Modbus 采集与 MQTT 通信永远可用

- **断网续传按优先级补传**：MQTT 断连 → SQLite offline_cache 本地缓存 → 恢复后报警优先补传 QoS1，保障极端工业现场数据零丢失

## 与旧方案的关系

| 旧方案内容 | 处理方式 |
|-----------|---------|
| TLV 协议（TlvProtocol.h、Session 状态机） | **废弃**。旧方案的 TLV 用于外部 AI 节点 TCP 通信，新方案 AI 在板载引擎完成，不再需要外部 TCP TLV。 |
| TCP Server（epoll ET 北向通道） | **废弃**。旧方案中进程 A 的 TCP Server 接收 AI 节点报文，新方案 AI 由进程 E 本地完成，TCP Server 移除。 |
| TcpSouthbound 适配器实现 | **废弃**。被 ModbusRtuDriver/ModbusTcpDriver 替代。 |
| ModbusPoller、Modbus RTU | ✅ **直接复用**，逻辑不变 |
| InternalMessage + UDS | ✅ **直接复用**，增加 source_type=3(AI检测) 和 0x05(AI心跳) |
| 共享内存 Ping-Pong | ✅ **直接复用**，ShmBlock 增加 AI 字段 |
| SQLite 异步批量 | ✅ **直接复用** |
| MQTT 客户端封装 | ⚠️ **改造**，Topic 升级为 SparkplugB 命名，增加 OTA/配置下发主题 |
| MQTT RPC | ⚠️ **增强**，升级为完整的配置热加载 + 模型 OTA RPC |
| 断网续传 | ✅ **直接复用** |
| Watchdog | ✅ **直接复用**，多监控一个子进程 E |
| Qt 监控台 | ⚠️ **扩展**，增加 AI 指标面板 + AI 快照显示 |
| ai-inference-engine（方案中的外部 AI 服务） | **废弃**。由 gateway_engine（进程 E）替代。 |
| 模拟 TCP 传感器（mock_sensor） | **废弃**。不再需要模拟外部 AI 节点 TCP 发包。 |
| 1000 并发压测脚本 | ⚠️ **保留框架**，压测目标改为多路 RTSP + Modbus 并发 |
