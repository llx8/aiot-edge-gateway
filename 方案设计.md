# 端-边-云 AIoT 工业边缘计算系统 — 技术方案

## 系统架构全景

两个物理隔离的子项目构成"端-边-云"闭环：

```
┌─────────────────────────────────────────────────────────────┐
│                      云端 MQTT Broker                        │
│                  QoS1 + Last Will / Retain                   │
└──────────────────────┬──────────────────────────────────────┘
                       │
    ┌──────────────────┴──────────────────┐
    │                                     │
    ▼                                     ▼
┌───────────────┐                ┌──────────────────────────────┐
│  项目二 (端侧)  │   TCP+TLV     │  项目一 (边缘侧)              │
│  Android 手机   │ ─────────────→│  i.MX6ULL / Linux PC        │
│  AIoT 视觉节点   │  报警报文      │  IoT 边缘网关                │
│               │                │                              │
│  CameraX → JNI │               │  进程A: 接入网关 (epoll ET)    │
│  NCNN YOLOv8n  │               │  进程B: 业务中枢 (规则/存储/云) │
│  无锁推理流水线 │               │  进程C: Qt 监控台 (可视化)     │
└───────────────┘                └──────────────────────────────┘
```

**核心架构原则**：端侧做特征提取（只传报警结论，不传原始视频），边缘侧做汇聚+决策+上云+可视化。

## 项目一：IoT 边缘网关 (aiot-edge-gateway)

运行于 i.MX6ULL（Cortex-A7, Linux）/ Linux PC。三进程多进程解耦，IPC 通信。

### 进程 A：接入网关 (gateway_access)

**职责**：网关的"南向接入层"，负责异构协议接入与归一化。双通道设计：

- **北向通道（TCP Server）**：epoll ET 模式接收 AI 视觉节点上报的 TLV 报文，协议解析后通过 UDS 转发给中枢进程。
- **南向通道（Modbus Poller）**：主动轮询 RS485 总线上的传统 Modbus RTU 传感器（温湿度计、智能电表等），读取寄存器数据后在网关内部封装为 TLV 报文，同样通过 UDS 转发。

**核心设计原则**：所有协议差异在进程 A 内部消化。对进程 B 和 Qt 界面来说，数据来源透明——它们只认 TLV。

**TLV 报文协议**

```
字节:   0       1       2       3    ...   N-2    N-1
    ┌───────┬───────┬───────┬───────┬─────┬───────┬───────┐
    │ magic │  ver  │total_len(2B)│type │ Value...│ CRC16 │
    │ 0x5A  │ 0x5A │   1   │  高/低  │     │(变长)  │(2字节)│
    └───────┴───────┴───────┴───────┴─────┴───────┴───────┘
      Header 6 字节                     Body       Trailer
```

```cpp
#pragma pack(push, 1)
struct TlvHeader {
    uint16_t magic;       // 0x5A5A, 网络字节序(大端)
    uint8_t  version;     // 1
    uint16_t total_len;   // Header + Value + CRC 总长度, 网络字节序
    uint8_t  type;        // 0x01=温湿度 0x02=开关量 0x03=模拟量 0x04=视觉告警 0xFF=心跳
};
#pragma pack(pop)
```

**CRC16**：CRC16/MODBUS，校验范围 Header + Value，不含尾部的 CRC 字段自身。

**连接状态机**

```
                    EPOLLIN ┌──────────────────┐
        ┌──────────────────→│   CONN_IDLE       │
        │                   │   等待 magic 0x5A5A│
        │                   └────────┬──────────┘
        │       收到部分 Header       │ 找到 magic,
        │                            │ 读满 6 字节 Header
        │                   ┌────────┴──────────┐
        │  EPOLLIN          │ CONN_READING_HEADER│
        │  ←───────────────│ 缓冲区未凑齐 6 字节  │
        │                   └───────────────────┘
        │                            │ Header 已齐, total_len 已知
        │                            │ 已收 < total_len
        │                   ┌────────┴──────────┐
        │         EPOLLIN   │ CONN_READING_BODY  │
        │        ←──────────│ 缓冲区 < total_len  │
        │                   └────────┬──────────┘
        │                            │ 已收 >= total_len
        │                   ┌────────┴──────────┐
        │                   │    解包 + CRC校验   │
        │                   │  校验通过→UDS转发   │
        │                   │  指针前移, 找下个包 │
        │                   └────────┬──────────┘
        │                            │
        └────────────────────────────┘
```

每个 TCP 连接绑定一个 `RingBuffer`（环形缓冲区）。epoll ET 下 `while (recv) until EAGAIN` 读完内核缓冲区，追加到 RingBuffer，然后驱动上述状态机找包边界。CRC 不通过则丢弃整包并计数。

**UDS 转发路径**：`/tmp/gateway_data.sock`，SOCK_STREAM。

> ⚠️ **UDS 粘包警告**：`SOCK_STREAM` 是流式协议，**没有消息边界**。进程 A 连续发两个 TLV 包，进程 B 一次 `recv` 可能收到粘包或半包。进程 B 接收 UDS 数据时，**必须复用进程 A 的 TLV 状态机 + RingBuffer 逻辑**，不能假设一次 `recv` 就是一个完整包。或者改用 `SOCK_SEQPACKET`（保留消息边界的本地套接字）。

> ⚠️ **UDS 重连机制**：多进程启动有先后——进程 B 可能还在初始化 SQLite/MQTT，进程 A 已经去 `connect()` UDS 了。**永远不要假设对方先启动好**。UDS Client 必须实现指数退避重试：`connect` 失败 → 等 100ms 重试 → 200ms → 400ms → ... → 最多 2s 间隔，最多重试 30 次。

### 南向采集模块：Modbus RTU 接入（进程 A 内部）

**为什么需要 Modbus**：真实的工业现场不仅有高端 AI 摄像头，还存在大量便宜、老旧的 RS485 串口传感器（温湿度计、智能电表、水泵控制器）。这些设备只支持 Modbus RTU 协议。只有同时接入两种协议，网关才真正具备"工业边缘网关"的价值。

**架构位置**：Modbus Poller 与 TCP Server 同在进程 A 内部，作为南向采集通道。

```
┌──────────────────────────────────────────────────────────┐
│  进程 A：接入网关 (gateway_access)                        │
│                                                          │
│  [北向] TCP Server (epoll ET) ← 接收 AI 节点 TLV        │
│  [南向] Modbus Poller (独立线程) → 轮询传统传感器        │
│               ↓ (读取寄存器)                             │
│         协议归一化 (封装为 TLV type=0x01)                │
│               ↓ (通过 eventfd 通知主线程)                │
│  [内部] UDS Client → 发送给进程 B                        │
└──────────────────────────────────────────────────────────┘
```

#### Modbus RTU 帧结构

Modbus RTU 帧非常简单，适合手写解析器展现底层功底：

```
┌──────────┬──────────┬───────────────┬──────────┐
│ 地址 1B  │ 功能码 1B│ 数据 N 字节    │ CRC16 2B │
│ (1~247)  │ 0x03=读 │(寄存器地址+数量)│          │
└──────────┴──────────┴───────────────┴──────────┘
```

常用功能码（工业温度/电表传感器仅需两个）：

| 功能码 | 名称 | 用途 |
|--------|------|------|
| `0x03` | Read Holding Registers | 读温度/湿度/电压值 |
| `0x06` | Write Single Register | 写阈值/配置参数 |

#### 轮询调度器设计（面试核心考点）

RS485 是半双工总线，同一时刻只能有一个设备说话。必须实现基于时间片的轮询调度器：

```
调度循环:
  for each device in device_list:
    1. 发送读请求 (功能码 0x03, 指定寄存器地址和数量)
    2. 阻塞等待响应 (配置 VMIN/VTIME 超时，如 500ms)
    3. 收到完整响应:
       - 校验 CRC16（复用 common/TlvProtocol.h 的 crc16_modbus）
       - 解析寄存器值，还原为物理量（温度/湿度/电量）
       - 协议归一化：封装为 TLV 报文 (type=0x01)
       - 通过 eventfd 通知主 epoll 线程 → UDS 转发
    4. 超时:
       - 累计超时次数
       - 连续 3 次超时 → 挂起该设备（避免拖慢整条总线）
       - 被挂起设备每 30s 尝试复活一次
```

#### 线程模型：Modbus 线程 + eventfd 桥接

Modbus 串口读取是阻塞的，不能放在 epoll 主循环中。解决方案：

```
ModbusThread (独立线程)                  主 epoll 线程
    │                                        │
    ├─ while(running):                       │
    │    ├─ 轮询设备 0                        │
    │    ├─ 轮询设备 1                        │
    │    ├─ ...                               │
    │    ├─ 封装 TLV                          │
    │    └─ write(eventfd, &cnt, 8) ─────→  epoll 监听到 eventfd
    │                                        ├─ read(eventfd) 消费事件
    │                                        ├─ 从无锁队列取出 TLV 包
    │                                        └─ write(uds_fd, tlv_packet)
```

**关键点**：eventfd 是 Linux 轻量级事件通知机制（8 字节计数器），可被 epoll 监听。Modbus 线程写 eventfd 唤醒主循环，零锁零竞争。

### 进程 B：业务中枢 (gateway_core)

**职责**：收接入进程报文、规则引擎、存储、上云、写共享内存。

**自身 epoll 循环**，监听四个 fd：
- UDS server fd（进程 A 报文流入，需复用 TLV 状态机 + RingBuffer 防粘包）
- MQTT 内部 socket fd
- eventfd（MQTT 回调线程唤醒主线程用）
- 信号 fd（SIGINT/SIGTERM 优雅退出）

> ⚠️ **MQTT 回调线程隔离**：paho-mqtt-cpp 的 `onDisconnected`、`messageArrived` 等回调在其**内部网络线程**中执行。回调里**禁止直接写 SQLite、禁止做 UDS 转发**——只能把事件推入无锁队列 + `write(eventfd)` 唤醒主 epoll 线程，由主线程统一调度。否则会阻塞 MQTT 心跳导致被 Broker 踢下线。

**规则引擎**

| 规则 | 条件 | 动作 |
|------|------|------|
| 高温报警 | 温度 > 80°C | SQLite + MQTT QoS1 + 共享内存 alarm_active=1 |
| 视觉入侵 | type=0x04 | SQLite + MQTT QoS1 + fsync 强制刷盘 |
| 传感器离线 | 30s 无心跳 | MQTT 发布离线告警 |
| 周期存储 | 每 10s | 批量写 SQLite |

**SQLite3 — 单线程异步批量落盘（防锁表设计）**

**为什么不能直接多线程写 SQLite**：规则引擎（处理报警）、Modbus 采集（周期数据）、周期存储定时器可能同时触发 `sqlite3_exec`。即使 WAL 模式允许一写多读，写操作本身依然是串行的——多个线程同时写会导致 `SQLITE_BUSY` 或线程阻塞，拖慢核心 epoll 循环。

**解决方案**：独立 DB 线程 + 内存队列 + 事务批量提交。

```
业务线程 (epoll)                    DB_Thread (独立)
    │                                    │
    ├─ 规则引擎触发报警                    │
    │  └─ push(sensor_record) ──────→ 无锁 SPSC 队列
    │                                    │
    ├─ Modbus 周期采集完成                 │
    │  └─ push(sensor_record) ──────→     │
    │                                    ├─ 每 100ms 或 攒够 50 条
    │                                    ├─ BEGIN TRANSACTION
    │                                    ├─ INSERT INTO sensor_data (...)
    │                                    ├─ INSERT INTO alarm_log (...)
    │                                    ├─ COMMIT
    │                                    └─ 关键报警: fsync() 强制刷盘
```

**关键设计**：
- 业务线程（epoll 主循环）**零阻塞**——只做 `push` 到无锁队列，不碰 SQLite
- DB 线程独占 SQLite 连接，从根本上消除锁竞争
- 批量事务提交，将磁盘 IOPS 降低 ~90%（100 条数据 1 次 fsync vs 100 次 fsync）
- 队列满时策略：丢弃最旧的非报警数据（报警数据优先级最高，永不丢弃）

```sql
PRAGMA journal_mode=WAL;
PRAGMA synchronous=NORMAL;   -- WAL+NORMAL 保证 DB 文件不损坏，写入性能最大化
                              -- 关键报警：事务 COMMIT 后手动 fsync() 强制刷盘

CREATE TABLE sensor_data (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    ts         INTEGER NOT NULL,
    node_id    INTEGER NOT NULL,
    type       INTEGER NOT NULL,
    value      REAL    NOT NULL
);

CREATE TABLE alarm_log (
    id         INTEGER PRIMARY KEY AUTOINCREMENT,
    ts         INTEGER NOT NULL,
    alarm_type TEXT    NOT NULL,
    node_id    INTEGER NOT NULL,
    detail     TEXT    NOT NULL
);
```

**断网续传 (Offline Caching & Sync)**

**工业痛点**：工厂网络极不稳定，断网时 MQTT 发不出去，报警和关键数据就丢了——这是工业场景绝对不能接受的。

**实现机制**：

1. MQTT 断连时（`onDisconnected` 回调），规则引擎将原本上云的数据**重定向**写入 SQLite 的 `offline_cache` 表
2. MQTT 重连成功后，触发后台**异步补传任务**，分批（每批 100 条）从 `offline_cache` 读取 → PUBLISH → 成功后 DELETE
3. 补传使用 QoS1，确保至少一次送达
4. `offline_cache` 表设置最大行数（如 100000 条），超过后删除最旧的非报警记录，防止磁盘写爆

```sql
CREATE TABLE offline_cache (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    ts          INTEGER NOT NULL,
    topic       TEXT    NOT NULL,
    payload     TEXT    NOT NULL,
    priority    INTEGER NOT NULL DEFAULT 0,  -- 0=周期数据, 1=报警(优先补传)
    created_at  INTEGER NOT NULL
);
CREATE INDEX idx_offline_priority ON offline_cache(priority DESC, id ASC);
```

**面试话术**："实现了基于本地 SQLite 的断网续传机制。网络异常时数据自动降级本地缓存，网络恢复后基于 MQTT QoS1 按优先级异步补传（报警优先），保障了极端工业现场下核心数据的零丢失。"

**MQTT 远程 RPC：云端反向控制 (Cloud-to-Edge Control)**

**工业痛点**：设备卖到全国各地，如果修改 Modbus 轮询周期、调整报警阈值、更新从站地址表，难道要派人去现场改配置文件或重新烧录固件？

**实现机制**：网关订阅 `gateway/gw001/rpc` 主题，云端下发 JSON 指令实现远程热配置。

```
云端下发:
  Topic: gateway/gw001/rpc
  Payload: {"id":"rpc_001", "method":"set_modbus_interval", "params":{"interval_ms":2000}}

网关处理:
  1. 进程 B MQTT 回调收到消息
  2. 解析 JSON，校验 method 白名单
  3. 通过 UDS 查询接口转发给进程 A
  4. 进程 A 修改 ModbusPoller 的定时器周期（std::atomic + 条件变量）
  5. 进程 A 回复 ACK → 进程 B → MQTT 发布 gateway/gw001/rpc/ack
```

**支持的 RPC 指令**：

| method | 参数 | 目标进程 | 说明 |
|--------|------|---------|------|
| `set_modbus_interval` | `interval_ms` | 进程 A | 动态修改 Modbus 轮询间隔 |
| `set_alarm_threshold` | `sensor_type, max_value` | 进程 B | 动态修改报警阈值 |
| `add_modbus_device` | `addr, reg_addr, name` | 进程 A | 热添加 Modbus 从站 |
| `remove_modbus_device` | `addr` | 进程 A | 热移除 Modbus 从站 |
| `get_device_list` | — | 进程 A | 查询当前在线的所有设备 |
| `trigger_offline_sync` | — | 进程 B | 手动触发断网缓存补传 |

**安全措施**（分层防护，由浅入深）：

| 层级 | 措施 | 说明 |
|------|------|------|
| 基础 | method 白名单 | 拒绝未定义的指令，防止执行任意方法 |
| 基础 | `id` 防重放 | 请求-应答匹配，同一 `id` 只执行一次，拒绝重复指令 |
| 基础 | 参数范围校验 | `interval_ms` 限制 [500, 60000]，`max_value` 限制合理范围，防止恶意参数（如 interval=1ms）导致 CPU 满载 |
| 进阶 | HMAC-SHA256 签名 | Payload 附带 `sign = HMAC-SHA256(payload, device_secret_key)`，网关用预置密钥验签，确保指令来源可信（设备密钥在出厂时烧录） |
| 进阶 | TLS 加密传输 | MQTT Broker 启用 TLS，防止中间人窃听指令内容 |

**开发阶段**：Phase 3b 先实现白名单 + 参数校验 + id 防重放。HMAC 签名和 TLS 在商业化落地时加入。

**面试话术**："构建了基于 MQTT 的云边反向 RPC 通道，支持云端动态下发 JSON 指令修改边缘侧采集策略与报警阈值，实现了设备的远程热配置。考虑到公网环境下的指令伪造和重放攻击风险，我在 RPC 模块设计了分层安全机制——除了 method 白名单和基于 id 的防重放校验外，在商业化落地时会引入基于设备唯一密钥的 HMAC-SHA256 签名校验，确保只有合法的云端才能下发控制指令。"

---

**共享内存 — 无锁双缓冲 (Ping-Pong Buffer) 防数据撕裂**

**为什么 version 自增不够**：原方案用 `version++` 保证读写一致性，但 `ShmBlock` 内部包含多个字段（甚至后续扩展的历史数据数组），写端在逐字段更新时，读端可能恰好触发 eventfd 读到"改了一半"的脏数据——这就是**数据撕裂**。version 只能检测到"写完没写完"，不能阻止"读的时候正在写"。

**解决方案**：Ping-Pong 双缓冲 —— 写端永远只写"非当前读取区"，写完后原子切换指针。

```
共享内存布局:
┌─────────────────────────────────────────────┐
│  ShmHeader (固定偏移 0)                      │
│  ├── magic:      0x47574D4D                 │
│  ├── read_index: 0 或 1 (原子, 指向读端可用) │
│  └── padding                               │
├─────────────────────────────────────────────┤
│  ShmBlock buffer[0]  (Ping)                │
│  ├── uptime_sec, total_packets, ...        │
│  └── last_alarm[128]                       │
├─────────────────────────────────────────────┤
│  ShmBlock buffer[1]  (Pong)                │
│  └── (同上)                                 │
└─────────────────────────────────────────────┘
```

**写端（进程 B）**：
1. 读 `read_index`，确定当前读端正在用的是哪个 buffer
2. 向**另一个** buffer（`1 - read_index`）写入完整数据
3. 写完后，**原子交换** `read_index = 1 - read_index`
4. `write(eventfd, &cnt, 8)` 通知读端

**读端（进程 C / Qt）**：
1. eventfd 触发 → `QSocketNotifier` 唤醒
2. 读 `read_index`，定位当前读区 buffer
3. **标量字段（cpu_usage, online_nodes 等）**：直接读取，无需拷贝
4. **数组字段（历史数据 history_buffer[]）**：**必须深拷贝（memcpy）到本地 QVector**——因为 Qt 渲染是异步的，如果 UI 还没画完 QCustomPlot 曲线，写端就切换 `read_index` 并覆盖这块 buffer，Qt 就会读到脏数据甚至崩溃。拷贝后立即释放"读区占用"，让写端可以继续写

```cpp
// ⚠️ 共享内存结构体严禁使用 #pragma pack(1)！
// 原因：ARM 平台（i.MX6ULL）对非对齐地址执行 std::atomic 操作会触发 Bus Error 崩溃。
// 共享内存不需要节省带宽——让编译器自然对齐，保证 atomic 和 uint64_t 在 8 字节边界。

struct ShmHeader {
    uint64_t    magic;                   // 8 字节，自然对齐
    std::atomic<uint32_t> read_index;    // 4 字节，紧跟 magic 后自然对齐
    uint32_t    _padding[3];             // 手动填充，对齐到 32 字节边界
};
static_assert(sizeof(ShmHeader) == 32, "ShmHeader must be 32 bytes");

struct ShmBlock {
    int64_t     uptime_sec;              // 8 字节
    int32_t     total_packets;           // 4 字节
    int32_t     total_alarms;
    float       cpu_usage;               // 4 字节
    float       mem_usage;
    int32_t     online_nodes;
    int32_t     mqtt_connected;
    int32_t     alarm_active;
    char        last_alarm[128];         // 128 字节
    // 可扩展：历史数据数组等
};
// 无需 static_assert 精确大小——自然对齐下编译器保证正确
```

**关键设计**：
- `read_index` 用 `std::atomic<uint32_t>` + `memory_order_acquire/release` —— 确保 CPU 缓存一致性
- 双缓冲写端永远不改读端区 → **读写物理隔离**，从根本上消除撕裂
- **标量字段**：读端直接读取 buffer，零拷贝
- **数组字段（如 history_buffer[]）**：eventfd 触发时迅速 `memcpy` 到 Qt 线程本地 `QVector`，拷贝完成即释放读区控制权，防止 UI 异步渲染期间被写端覆盖
- 写端 `1 - read_index` 切换，只有一个写端（进程 B），无需 CAS 循环

**面试话术**："为了防止多进程读写共享内存时发生数据撕裂，我抛弃了简单的版本号机制，采用了无锁双缓冲（Ping-Pong）设计。写端在备用区组装完整数据，完成后原子切换读指针，读写物理隔离。对于标量数据，读端可以直接读取共享内存实现零拷贝；对于历史曲线等数组数据，我会在 eventfd 唤醒时迅速深拷贝到 Qt 线程本地容器，随后立即释放读区控制权，确保 UI 异步渲染期间数据绝对安全。"

### 进程看门狗：Watchdog Daemon（进程 D）

**工业痛点**：C++ 程序难免遇到野指针、栈溢出等导致 Segmentation Fault 崩溃。工业设备通常部署在无人值守的机房或野外，没有显示器和键盘，死机了不可能让工人去敲命令重启。

**设计方案**：引入第四个进程——Watchdog 守护进程，由它统一管理 A、B、C 三个子进程的生命周期。

```
┌─────────────────────────────────────────────┐
│  gateway_watchdog (进程 D, 父进程)           │
│                                             │
│  ├── fork() → gateway_access (子进程 A)      │
│  ├── fork() → gateway_core   (子进程 B)      │
│  ├── fork() → gateway_monitor (子进程 C)      │
│                                             │
│  ├── waitpid(-1, &status, WNOHANG) 循环监听   │
│  │   ├── 子进程正常退出 → 记录日志, 重新 fork  │
│  │   └── 子进程异常崩溃 → 记录 Crash 日志,     │
│  │       等 1s 后重新 fork 拉起               │
│  │                                           │
│  ├── SIGINT/SIGTERM → 转发给所有子进程 → 优雅退出│
│  └── 自身被 systemd 管理（Restart=always）     │
└─────────────────────────────────────────────┘
```

**关键策略**：

| 策略 | 说明 |
|------|------|
| 崩溃重拉 | 子进程异常退出后等待 1s 再拉起，防止疯狂崩溃-重启循环 |
| 重启退避 | 如果 30s 内重启超过 5 次，暂停重启并记录 FATAL 日志（防止硬件故障导致的无限重启） |
| 依赖顺序 | 先启动 B（core），再启动 A（access），最后启动 C（monitor）——启动间隔各 500ms |
| 日志隔离 | 每个子进程 stdout/stderr 重定向到独立日志文件，Watchdog 自身写 watchdog.log |
| systemd 兜底 | Watchdog 自身配置 `Restart=always`，即使 Watchdog 挂了也能被 systemd 拉起 |

**简化的 bash 实现**（开发阶段可用，不必写 C++ Watchdog）：

```bash
#!/bin/bash
# scripts/watchdog.sh
trap 'kill $(jobs -p); exit 0' SIGINT SIGTERM

restart_process() {
    local name=$1 cmd=$2
    while true; do
        echo "[watchdog] Starting $name..."
        $cmd &
        local pid=$!
        wait $pid
        local code=$?
        echo "[watchdog] $name (pid=$pid) exited with code=$code, restarting in 1s..."
        sleep 1
    done
}

restart_process "gateway_core"    "./build/bin/gateway_core" &
restart_process "gateway_access"  "./build/bin/gateway_access --modbus-port /tmp/vcom_gateway" &
restart_process "gateway_monitor" "./build/bin/gateway_monitor" &
wait
```

**面试话术**："设计了基于 fork/waitpid 的多进程看门狗（Watchdog）机制，配合指数退避重启策略与 systemd 双重兜底，实现子进程崩溃后的秒级自动拉起与故障隔离，满足工业现场 7×24 小时无人值守的高可用需求。"

## 开发路线：洋葱模型（4 个里程碑）

**核心思想**：由内向外剥洋葱——先跑通"接入→存储→展示"最短链路，再逐层加上工业协议、云边协同、高可用铠甲。每个里程碑都是可独立 demo 的完整闭环。

```
         ┌──────────────────────────────────┐
         │  M4: 高可用铠甲 (压测/Watchdog)    │
         │  ┌────────────────────────────┐  │
         │  │ M3: 工业灵魂 (Modbus/MQTT)  │  │
         │  │ ┌──────────────────────┐  │  │
         │  │ │ M2: 核心骨架 (MVP)   │  │  │
         │  │ │ TCP→UDS→SQLite→Qt   │  │  │
         │  │ └──────────────────────┘  │  │
         │  └────────────────────────────┘  │
         └──────────────────────────────────┘
```

## 🧱 防重构三暗桩（MVP 阶段就必须埋好）

> 这些不是"以后优化"——是**第一天写 MVP 就要做的架构决策**。不做的话，里程碑 2 加 Modbus 时就得拆进程 B 的代码。

### 暗桩 1：消息总线化（UDS 统一内部格式）

**❌ 错误做法**：进程 A 把 TCP 收到的原始字节流直接 `write(uds_fd)`，进程 B 自己去解析 TLV。
**✅ 正确做法**：进程 A 在内部解析好 TLV，通过 UDS 发送**统一内部结构体**，进程 B 只认这一种格式。

```cpp
// common/InternalMessage.h —— UDS 唯一传输格式
struct InternalMessage {
    int32_t  source_type;    // 0=TCP, 1=Modbus, 以后加 CAN=2
    int32_t  node_id;        // 来源节点 ID
    uint8_t  tlv_type;       // 0x01=温湿度, 0x04=视觉告警, 0xFF=心跳
    std::vector<uint8_t> payload;  // TLV value 字段
};
```

**好处**：以后加 Modbus 时，进程 A 只需把 Modbus 数据也塞进 `InternalMessage`，进程 B **一行代码都不用改**。

### 暗桩 2：南向适配器接口（ISouthbound）

MVP 阶段虽然只有 TCP，但**第一天就要建接口**：

```cpp
// common/ISouthbound.h
class ISouthbound {
public:
    virtual ~ISouthbound() = default;
    virtual void start() = 0;
    virtual void stop() = 0;
    virtual std::string_view name() const = 0;

    using DataCallback = std::function<void(const InternalMessage&)>;
    void set_data_callback(DataCallback cb) { m_on_data = std::move(cb); }

protected:
    DataCallback m_on_data;  // 子类采集到数据后调用此回调
};
```

- MVP 阶段：写一个 `TcpSouthbound` 实现它
- 里程碑 2：写一个 `ModbusSouthbound` 实现它，主流程零改动

### 暗桩 3：配置驱动（拒绝 Hardcode）

从第一天起，所有可变参数从配置文件读取，**代码里不出现硬编码的 IP/端口/路径**：

```ini
# conf/gateway.conf
[tcp]
port = 9000
max_connections = 1024

[uds]
data_path = /tmp/gateway_data.sock
query_path = /tmp/gateway_query.sock

[sqlite]
db_path = /var/lib/gateway/gateway.db

[mqtt]
broker = 192.168.1.100
port = 1883
client_id = gateway/gw001

[modbus]
port = /tmp/vcom_gateway
baudrate = 9600
poll_interval_ms = 2000
```

**好处**：里程碑 3 加 RPC 远程修改配置时，只需要修改内存中的配置对象并 reload，不需要改核心逻辑。

## 简历写法

**项目名称**：多协议 AIoT 工业边缘计算网关与监控系统

**技术栈**：`C++17/20` `epoll` `UDS` `共享内存` `Qt5` `SQLite3` `MQTT` `Modbus RTU` `进程管理` `ARM Linux` `NCNN` `Android NDK`

**项目描述**：
> 针对工业现场异构设备共存、网络不稳定、无人值守三大痛点，自研"端-边-云"多协议高可用边缘网关。网关南向支持 TCP 自定义协议（AI 视觉节点）与 RS485 Modbus RTU（传统传感器）双通道接入，内部完成协议归一化（统一 TLV 报文）；北向 MQTT 云边协同，含断网续传与远程 RPC 反向控制；四进程架构配合 Watchdog 看门狗实现 7×24 无人值守自愈。

**技术亮点**：
- **异构协议接入与归一化**：设计南向采集适配器接口（Adapter Pattern），基于 termios 手写 Modbus RTU 协议栈（支持 0x03/0x06 功能码，复用 CRC16/MODBUS）；实现 RS485 半双工总线轮询调度 + 超时剔除/复活机制，Modbus 数据在网关内部归一化为 TLV，上层业务零感知
- **无锁双缓冲共享内存**：针对多进程读写数据撕裂问题，采用 Ping-Pong Buffer 设计，写端在备用区组装数据后原子切换读指针，读写物理隔离；标量字段读端零拷贝，数组字段深拷贝到本地容器后释放读区，防止 Qt 异步渲染期间数据被覆盖
- **异步批量 SQLite 落盘**：独立 DB 线程 + 无锁 SPSC 队列 + 事务批量提交，业务线程零阻塞，磁盘 IOPS 降低 ~90%；WAL 模式 + 报警 fsync 强制刷盘确保断电不丢
- **断网续传**：MQTT 断连时数据自动降级写入本地 SQLite `offline_cache` 表，恢复后按优先级（报警 > 周期数据）异步补传 + QoS1 保障，极端工业现场数据零丢失
- **MQTT 远程 RPC**：基于 JSON 的云边反向控制通道，支持远程热修改轮询周期、报警阈值、动态添加/移除 Modbus 从站，免去现场运维成本
- **进程看门狗**：fork/waitpid 多进程守护 + 指数退避重启策略 + systemd 双重兜底，子进程崩溃秒级自动拉起，满足 7×24 无人值守高可用
- epoll ET 高并发 + 自定 TLV 二进制协议 + 环形缓冲区粘包/半包防御
- Modbus 阻塞串口与 epoll 非阻塞主循环的线程分离 + eventfd 异步桥接
- MQTT Last Will 遗嘱机制，网关断电 Broker 秒级感知离线
- socat 虚拟串口 + Python pymodbus 零成本闭环测试（无需 RS485 硬件）
